Pulse / Architecture

How Pulse works

A text message comes in. 22 scrapers, an agent brain with tool calling, and a 480-character limit later, a recommendation goes out.

Overview

Text in, picks out

Pulse is an SMS-based AI assistant that recommends NYC nightlife and events. A user texts a neighborhood name and gets back curated picks within seconds. The entire experience happens in a single SMS thread — no app, no account, no links until asked.

Behind the scenes, Pulse scrapes 22 event sources every morning, deduplicates and caches them in memory, then routes every message through checkMechanical for $0 shortcuts (bare numbers, "more", "help", greetings) and the agent brain (Gemini 2.5 Flash Lite tool calling, ~$0.0005/msg) for everything else. The agent brain uses three structured tools — search_events, get_details, respond — to understand intent, pick events, and compose the SMS.

Walkthrough

What happens when you text "williamsburg"

Let's trace a single message through the entire system. A user with no session history texts "williamsburg" at 7:14 PM on a Friday.

Step 1 Guard rails <1ms, $0

handler.js receives the Twilio webhook. Three checks run before any processing:

  • TCPA check — Is this "STOP", "UNSUBSCRIBE", "CANCEL", or "QUIT"? If yes, drop silently (no reply). Twilio manages the opt-out list; we just make sure no response leaks.
  • Dedup — Has this MessageSid been seen in the last 5 minutes? Twilio retries on timeout, so this prevents double-processing.
  • Budget — Has this phone number spent over $0.10 in AI costs today? If so, send a friendly limit message and stop. Pricing is provider-aware (Haiku input: $1/M tokens, Gemini: $0.10/M).

"williamsburg" passes all three. Cost so far: $0.00. Latency: <1ms.

Step 2 Routing — mechanical miss, agent brain ~2s, ~$0.0005

checkMechanical in agent-brain.js checks the message against $0 shortcuts: help, bare numbers 1-5, "more", greetings, thanks, bye. "williamsburg" doesn't match any of these — it's a semantic request that needs the agent brain.

The message goes to callAgentBrain, which sends the message, session context, and event pool to Gemini 2.5 Flash Lite with three tools: search_events (neighborhood, categories, time, free, intent), get_details, and respond. The model calls search_events with the structured parameters it understands from the message.

Mechanical shortcuts handle a small percentage of messages at $0 cost. Everything semantic (neighborhoods, categories, vibes, compound queries, off-topic deflection) goes through the agent brain's tool calling.

Step 3 Tool execution & event fetching deterministic, $0

The agent brain calls search_events with structured parameters. The handler executes the tool deterministically:

  • Neighborhood resolution — The neighborhood param from the tool call resolves to "Williamsburg" via the 36-neighborhood map, including aliases and landmarks.
  • Filter state from tool params — Categories, time_filter, free_only, and date_range come directly from the search_events tool call parameters. These are structured and validated — the system of record for filter state.
  • Event pool — getEvents("Williamsburg") queries the in-memory cache, applies filters from the tool params, and returns matched events sorted by proximity.

This is the key architectural decision (P1): structured tool calls own state, free-text owns language. Session state is derived from the agent brain's tool call params, never parsed from free-text LLM output. The handler saves activeFilters as lastFilters from tool call params.

Step 4 Event fetching & curation 14 → 15 tagged events

getEvents("Williamsburg") queries the in-memory cache. The cache was built at 10am from 22 scrapers. Events are already deduped and sorted by proximity to Williamsburg's center coordinates.

Say it returns 14 events. Before entering the tagged pool, curation.js runs deterministic quality filters:

  • filterKidsEvents — Drops NYC Parks events that match children's patterns ("storytime", "toddler", "ages 3-5"). Without this, Claude would occasionally recommend a toddler puppet show alongside a DJ set.
  • filterIncomplete — Drops events below a completeness threshold (missing name, venue, or time). Prevents Claude from recommending "TBA at TBD".
  • validatePerennialActivity — Perennial picks (always-good bars/venues) are only included if they have a specific activity tonight (trivia, live jazz, DJ). Removes generic "nice bar" suggestions that don't answer "what's happening tonight?"

After curation: 12 events remain. Plus 2 perennial picks with tonight activities. The curated events are serialized with enrichment fields (source_vibe, venue_size, interaction_format, recurring) and returned to the agent brain for SMS composition. For this filterless first message, the brain picks freely from the full pool.

Step 5 SMS composition ~1.5s, lightweight prompt

brainCompose takes the matched events and writes the SMS. It uses a lightweight ~400-token system prompt (BRAIN_COMPOSE_SYSTEM) focused on voice, format, and the 480-char limit. The compose prompt includes source vibe tiers so the model prefers discovery/niche picks over mainstream platforms.

The compose call returns the SMS text directly — no JSON parsing needed. The agent brain architecture separates reasoning (tool calling step) from rendering (compose step), keeping each focused and cheap.

Cost: ~$0.0005 total for both steps. Latency: ~2-3s end-to-end.

Step 7 Send, save & persist filters 289 of 480 chars

The SMS text (289 characters, under the 480-char cap) is sent via Twilio. The handler saves the session atomically via saveResponseFrame() — picks, event map, neighborhood, and active filters (derived from the tool call params). A JSONL trace captures the full request for evals.

Now when the user texts "2", checkMechanical matches it as a details request, looks up pick #2 (Baby's All Right) from session.lastPicks, and sends the full event details with a link. No AI call needed ($0).

When they text "how about comedy", the agent brain calls search_events({ neighborhood: "williamsburg", categories: ["comedy"] }). The handler executes the tool, fetches comedy events, and brainCompose writes the SMS (~$0.0005).

Fallback

If Gemini returns a 503, the agent brain falls back to Anthropic Claude Haiku for the same tool-calling flow. The fallback is transparent — same tools, same handler logic, different provider.

Data Pipeline

Daily scrape, in-memory cache

Every morning at 10am ET, Pulse runs all 22 scrapers in parallel. Each scraper extracts structured event data from its source — parsing HTML, hitting APIs, or using Claude to extract from newsletters. Events are deduped across sources (hashed by name + venue + date), ranked by source trust weight, and cached in memory. No scraping happens in the hot request path.

Not every layer runs on every request — "details" only needs the session, "more" reuses the existing pool.

Structured tool calls own state

Filter state (categories, time_filter, free_only, date_range) comes from the agent brain's search_events tool call parameters — structured and validated. The handler saves activeFilters as lastFilters from tool call params via saveResponseFrame. The LLM never reports state in free text — only in structured tool calls.

Filters persist across turns because the agent brain sees session context and can include or drop filters in each new search_events call. Intent types (new_search, refine, pivot) control whether neighborhood switches preserve existing filters.

Sources

22 event sources, ranked by trust

Events are deduplicated across sources using a hash of name + venue + date. When the same event appears in multiple sources, the version from the higher-trust source wins. Sources with geo data (lat/lng) also teach venue coordinates to the shared venue map. Each source belongs to a vibe tier — discovery (indie newsletters, curated blogs), niche (single-venue or specialized), platform (structured aggregators), or mainstream (large ticketing platforms).

Sessions

Multi-turn conversation state

Each phone number gets a session that persists for 2 hours. The session tracks what the user has been shown, what filters they've applied, and what neighborhoods they've visited. This lets follow-up messages like "how about comedy" or "later tonight" modify the active context rather than starting over.

Agent Brain

Tool calling + lightweight compose

The agent brain splits reasoning from rendering. Step 1: Gemini 2.5 Flash Lite receives the user message, session context, and available tools, then calls the appropriate tool with structured parameters. Step 2: brainCompose takes the matched events and writes the SMS using a lightweight ~400-token system prompt focused on voice, format, and the 480-char limit.

This architecture replaced the previous unified LLM + composable skills approach. Tool calling gives us structured state (P1) without parsing free text, and the lightweight compose prompt avoids the over-filtering problems that conditional skills were designed to solve.

search_events

The primary tool. Accepts neighborhood, categories (array for multi-category OR), time_filter, free_only, date_range, and intent (new_search, refine, pivot). The handler executes the search deterministically against the event cache.

get_details

Returns full details for a specific event by pick number. Used when the user asks about a specific pick ("tell me more about 2").

respond

For conversational responses that don't need event data — greetings, off-topic deflection, clarification. Returns text directly without searching events.

brainCompose

Not a tool — a separate Gemini call after tool execution. Receives the matched events with enrichment fields (source_vibe, venue_size, interaction_format) and writes the SMS text. ~400-token system prompt focused on voice, format, and 480-char limit.

Why tool calling over monolithic prompts?

The previous unified LLM architecture used a single Claude Haiku call with 12 conditional skill modules to handle intent, event selection, and SMS composition in one pass. Tool calling separates these concerns: the brain figures out what to search for (structured params), the handler executes the search (deterministic), and the compose step writes the SMS (lightweight prompt). This eliminates filter-state parsing from free text (P1), reduces prompt complexity, and halves the per-message cost.

Quality

5-layer eval system

Quality is measured offline against stored request traces. The eval system runs in layers, from fast deterministic checks to expensive LLM judges. Each layer catches different failure modes.

  • 1 Unit tests — Pure function tests (formatters, geo, neighborhoods). Fast, no API calls. Run on every push.
  • 2 Code evals — 24 deterministic trace checks: 480-char limit, valid intent, valid neighborhood, picked events exist, filter match alignment, time filter accuracy, discovery lean, and more. Free and instant. (evals/code-evals.js)
  • 3 Extraction audit — Tier 1: verify Claude-extracted evidence quotes appear verbatim in source text. Tier 2: optional LLM judge for semantic accuracy. (evals/extraction-audit.js)
  • 4 Pipeline evals — Multi-turn scenario tests with expected outcomes. Trace assertions check intent, neighborhood, event presence, and banned phrases. (scripts/run-scenario-evals.js)
  • 5 LLM judges + A/B — Tone judge ("sounds like a friend, not a bot?"), relevance judge, and model-vs-model preference tests. Expensive, run on demand. (evals/judge-evals.js, scripts/run-ab-eval.js)
# Run the eval layers $ npm run eval # Layer 2: code evals on stored traces $ npm run eval:judges # Layers 2+5: code evals + LLM judges $ node scripts/run-scenario-evals.js # Layer 4: multi-turn scenarios $ node scripts/run-ab-eval.js # Layer 5: A/B comparison

Observability

Traces, alerts, dashboards

Every request writes a structured trace. Health monitoring catches scrape failures and slow responses. Three dashboards provide real-time visibility.

Request traces

Per-request JSONL records capture intent, neighborhood, picked events, AI costs, timing, and the full SMS response. Daily rotation with 4-file max. A 200-entry ring buffer powers the eval UI.

Email alerts

Health alerts fire on scrape failures (6hr cooldown). Runtime alerts fire on slow responses or errors (30min per-type cooldown). Sent via Resend. No-op if API key is unset.

Health dashboard

Per-source timing, success/failure status, history sparklines, extraction quality metrics. Available at /health.

Cost tracking

Per-user daily budget of $0.10. Provider-aware pricing (Haiku vs Gemini). Budget resets daily on NYC timezone. Friendly message when exceeded.
