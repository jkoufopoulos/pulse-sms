How the Tool Calling Loop Works
Haiku thinks → calls a tool → gets result → thinks again (or writes the SMS). If over 480 chars, model rewrites shorter.
Example: User texts "Bushwick"
Haiku Brain
"what should I do?"
Execute Tool
search / respond / compose / welcome
Tool Result
events found, formatted data
SMS!
Step-by-step for "Bushwick"
1
User message arrives
{ Body: "Bushwick", From: "+15551234567" }
Passes guard + mechanical check (not "help" or "STOP")
2
Turn 1 — Model calls search_events
search_events({ neighborhood: "bushwick", intent: "new_search" })
brain-execute.js builds pool → curatePool scores by interestingness → top 10 events + nearby highlight
3
Tool result fed back to model
{ events: [10 curated picks], nearby_highlight: { hood: "Williamsburg", reason: "16 editor's picks" } }
Model sees curated events + nearby signal, picks 1-3, writes SMS with optional tease
4
Turn 2 — Model writes plain text (no tool call = done)
"Bushwick tonight — Mood Ring has a vinyl night, tiny room, good speakers, low-key vibe. Or there's a comedy open mic at Houdini Kitchen if you want something more interactive. Which sounds more like your night?"
Contrasting picks pattern: two different vibes, user self-selects. If >480 chars → rewrite loop → smartTruncate → Twilio sends
How follow-ups use the same loop (session carries context)
"how about comedy"
search_events({ neighborhood: "bushwick", categories: ["comedy"] }) → 3 comedy events → Model writes SMS
"2"
search_events({ intent: "details", pick_reference: "2" }) → full details for pick #2 → Model writes detail SMS
"later tonight"
search_events({ neighborhood: "bushwick", categories: ["comedy"], time_after: "22:00" }) → late-night comedy → Model writes SMS
"thanks!"
respond({ intent: "thanks", message: "..." }) → Model writes friendly goodbye
Key insight: The model decides what tool to call and what params to pass based on conversation context + session state. Neighborhood sticks from message to message. Filters accumulate ("comedy" + "late night") or reset ("forget the comedy"). Pool is pre-curated to top 10 by interestingness before the model sees it. Each event carries enrichment metadata (source_vibe, venue_size, scarcity, editorial_note) that the model uses to explain WHY a pick is interesting. The model presents contrasting picks (one chill, one active) instead of similar listings — the contrast IS the narrowing question. State is always derived from the structured tool call params — never parsed from the model's free-text output (Principle P1).

What's structurally different
1. Events get richer. Today each event carries metadata flags. In destination, events also carry venue personality ("sweaty dive bar, gets packed by 9") and serendipity scores. The agent doesn't need new tools — it just sees better data in the same pool.
2. The agent knows you. Today every user starts cold. In destination, the system prompt includes your persistent taste profile: neighborhoods you frequent, categories you pick, how you decide. The agent adapts without asking.
3. Pulse texts first. Today it's purely reactive: user texts, Pulse responds. In destination, a scheduler scans events against profiles and reaches out when there's a high-confidence match. One new path: cache × profiles → SMS.
