Pulse SMS Architecture

Pulse turns a text message into a curated night out. Users text a neighborhood — "Bushwick", "LES", "prospect park" — and get back 1-3 opinionated event picks under 480 characters. They can reply a number for details, "more" for extra picks, or refine with follow-ups like "how about comedy" or "later tonight." The entire experience happens in a single SMS thread with no app, no account, and no links until you ask.

Cross-Source Dedup

How it works: Event IDs are MD5 hashes of normalizedName|venue|date. The same event from Dice and BrooklynVegan gets the same ID. Sources are merged in weight order (Skint 0.9 first, Eventbrite 0.7 last), so the first write wins — higher-trust sources always take precedence.

All Prompts

Complete text of every system prompt in the system, with annotations.

1. EXTRACTION_PROMPT — Event Extraction ~1,600 tokens | Used by: Skint, Nonsense NYC, Yutori, Screen Slate | ai.js:extractEvents()
Role: Event Extractor for Pulse (NYC). Convert messy source text into normalized event records. RULES: - VENUES vs EVENTS: If a venue hosts a specific event, extract the EVENT with venue as venue_name. Only extract venue-only records when no specific event is mentioned. Bars, restaurants, game spots, arcades — extract these too with best-fit category. Permanent venues: set date_local/start_time_local to null, time_window to "evening", confidence to 0.6. - SOURCE URLs: Use [Source: URL] markers before each item as that event's source_url. Prefer per-item [Source: URL] over top-level source_url input. - TRUTH + SAFETY: Extract only what is explicitly present. Do not guess. If missing, set null. Prefer NYC interpretation (America/New_York). - DATE RESOLUTION: Use retrieved_at_nyc to resolve relative days. "today"/"tonight" → retrieved_at_nyc date. Past day names → set confidence to 0.1. - CONFIDENCE SCALE: 0.9+: name + date/time + location clearly present 0.7–0.85: name + (date OR time window) + partial location 0.4–0.65: name clear but time/location ambiguous < 0.4: too ambiguous → needs_review = true 0.1: event date has already passed - NEEDS_REVIEW TRIGGERS: confidence < 0.5, ambiguous name, unresolvable date, both venue_name and neighborhood missing, contradictory price info. - DEDUPE HINT: Output duplicates separately; downstream will dedupe. OUTPUT FORMAT: { "events": [{ source_name, source_url, name, description_short, venue_name, venue_address, neighborhood, latitude, longitude, category, subcategory, start_time_local, end_time_local, date_local, time_window, is_free, price_display, ticket_url, map_hint, confidence, needs_review, evidence: { name_quote, time_quote, location_quote, price_quote } }] } EXAMPLES: Skint-style newsletter, Screen Slate film listing, Nonsense NYC email
2. buildBrainSystemPrompt — Agent Loop System Prompt ~400 tokens | Used by: runAgentLoop() | brain-llm.js | Hot path
Role: Pulse — NYC nightlife SMS bot. Agent loop with 2 tools: search_events and respond. The model calls tools to fetch events or handle conversation, receives results, and writes the final SMS as plain text when ready. Multi-turn loop, max 3 iterations. Gemini 2.5 Flash Lite with tool calling. No separate compose step — the model writes natural prose SMS directly as its final text output.
3. DETAILS_SYSTEM — Event Details ~400 tokens | Used by: composeDetails() | ai.js:320 | Lightest prompt
Role: Pulse — NYC "plugged-in friend" texting about a spot you recommended. Write like a real person — warm, opinionated, concise. Never robotic. CONTENT PRIORITY (include in order, cut from bottom if long): 1. Vibe / what makes it worth going (LEAD with this) 2. Time (tonight at 9, doors at 10, etc.) 3. Price or "free" 4. URL (always include if provided) 5. Address (only if space remains) CONSTRAINTS: - 320 character limit (SMS split prevention) - Plain text only. No JSON, no quotes, no preamble. - No list format or bullet points. One natural paragraph like a text from a friend. - No Yelp URLs.
Agent loop prompt: buildBrainSystemPrompt() lives in brain-llm.js (not exported from prompts.js). The agent loop uses Gemini 2.5 Flash Lite with 2 tools: search_events and respond. The model writes the SMS as plain text — no separate compose step. Falls back to Anthropic Haiku on Gemini failure.
prompts.js exports: EXTRACTION_PROMPT, DETAILS_SYSTEM. The UNIFIED_SYSTEM prompt remains in prompts.js but is no longer used in the hot path — the agent loop has its own inline prompt via buildBrainSystemPrompt.

Where Claude Adds Value

Pulse uses Gemini/Claude for exactly 3 things. Each one is chosen because code can't do it well — and LLM-specific strengths (language understanding, taste, tone, messy-input parsing) are the right tool.

LLM's 3 Jobs

1. Extraction — Messy HTML to Structured Data

ai.js:extractEvents() | EXTRACTION_PROMPT | Scrape pipeline

4 sources (Skint, Nonsense NYC, Yutori, Screen Slate) publish events as newsletters fetched via Gmail API. No JSON-LD, no API, no consistent structure. Claude reads the raw text and produces normalized event records with dates, venues, times, prices, and confidence scores.

Why Claude: These sources have wildly varying formats that change without notice. Regex/Cheerio would require per-source maintenance and break on every redesign. Claude handles format drift gracefully — it reads like a person would.

Why not code: Code-based parsing is brittle for unstructured newsletters. The Skint mixes free-form prose with event listings; Nonsense NYC uses an email newsletter format with no consistent HTML structure; Screen Slate splits by venue headers with free-form film listings. Rule-based extraction would be a maintenance nightmare.

2. Agent Loop — Intent Understanding + Curation + SMS Writing

agent-loop.js:handleAgentRequest() → llm.js:runAgentLoop() | Gemini 2.5 Flash Lite | SMS pipeline (all messages except help + TCPA)

The agent loop handles everything: understanding intent via tool calling (search_events, respond), curating picks from the event pool, and writing the final SMS as natural prose — all in one multi-turn loop (max 3 iterations). Tool call params carry structured state: neighborhood, categories, time_filter, free_only, date_range. No separate compose step — the model writes plain text SMS directly.

Why LLM: These messages require understanding context + intent + geography simultaneously. "village" could be East Village, West Village, or Greenwich Village depending on context. "got jazz?" with a Bushwick session means "jazz near Bushwick" not "jazz anywhere." The model also writes opinionated, personality-driven SMS — "legendary basement spot, incredible players" vs "Jazz at Smalls Jazz Club, 9:30pm, $20." The personality IS the product.

Why not code: NLP for open-ended SMS messages with slang, abbreviations, and implicit session context is where statistical matching fails. The agent loop's tool calling gives structured params (safe for state) while using LLM semantic understanding for interpretation. And code can't write with taste and voice.

3. Details Writing — Natural Language About a Venue

ai.js:composeDetails() | DETAILS_SYSTEM | SMS pipeline (on demand)

When a user replies "1" to get details on a pick, Claude writes a natural-language description of the event: why it's worth going, the vibe, time, price, and a link. 320-char limit.

Why Claude: Same voice/tone benefit as the agent loop. A structured fallback exists (formatEventDetails()) but it reads like a database dump. Claude makes it feel like a friend giving you the rundown.

Why not code: There IS a code fallback — it's used when Claude fails. The code version works but lacks personality. The Claude version is the premium experience.

What Stays as Code (and Should)

Every other operation in Pulse is deterministic and runs without LLM calls. This is intentional — LLM calls add latency and cost. They should only be used where their strengths are irreplaceable.

Agent Loop Architecture

The agent loop replaces the old agent brain + brainCompose two-step. Instead of separate tool-calling and compose LLM calls, the agent loop uses multi-turn tool calling — the model calls tools, gets results, and writes the SMS as plain text in one continuous loop.

Before (agent brain + brainCompose)After (agent loop)
checkMechanical (6 patterns, $0) + callAgentBrain (Gemini tool calling) + brainCompose (Gemini compose) — two separate LLM steps checkMechanical (help + TCPA, $0) + handleAgentRequest → runAgentLoop (Gemini multi-turn tool calling, max 3 iterations) — one continuous loop, model writes SMS directly
State derived from tool call params, but compose was a separate step with its own prompt (BRAIN_COMPOSE_SYSTEM) State derived directly from tool call params (search_events args). Model writes plain text SMS — no separate compose step, no BRAIN_COMPOSE_SYSTEM.
Tool calling separates intent from execution, then a second LLM call composes the SMS Multi-turn loop: model calls tools → code executes them → results fed back → model decides next action or writes SMS. Each step is observable.
Design principle: The LLM's attention budget is finite. The agent loop keeps prompts small (~400 tokens) by delegating state to structured tool params. Deterministic curation (curation.js) still runs pre-compose — the LLM focuses on taste and writing. The model writes natural prose SMS, not numbered emoji lists.

Remaining Opportunities

Extraction prompt decomposition: EXTRACTION_PROMPT is the one remaining monolith (~1,600 tokens). It handles venue-vs-event distinction, date resolution, confidence scoring, and output format all in one. Could be decomposed per-source-type if extraction quality diverges across sources.
